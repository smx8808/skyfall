Lab 2 Details

Goal: Configure Cisco ASA as a remote access server

At the Center we have an ASA firewall (virtual) and outside and inside are unmanaged switches then a server inside and a workstation outside. you may want to add a management workstation connected to the management port of the ASA Firewall so you can experience similar to production access.

After you create your typology, save the lab, then select start the lab – may take upto 15 min to start all hosts – be patient.

      1. Design our Network and VPN Access twice and deploy once.

    2. We build and design our layout, ip addressing and determine ranges, interface addressing prior to configuring anything.

    One Caveat I will point out is that if you expect to use an (off-net) computer ie: one that is not in the simulator, like your desktop you will replace the desktopO with the “External Connector” looks like a cloud.

    3. Lets configure our interfaces

    !
    interface GigabitEthernet0/0
    nameif outside
    security-level 0
    ip address 10.100.100.1 255.255.255.0
    !
    interface GigabitEthernet0/1
    nameif inside
    security-level 100
    ip address 192.168.10.254 255.255.255.0
    !

    4. Next lets configure our IP Pool that VPN clients will be assigned when they connect as a remote VPN user, we will need to let the ASA know that we are using this pool for remote VPN users

    ip local pool VPN_POOL 192.168.100.100-192.168.100.200

    ASA0(config)# vpn-addr-assign local

    5. Under normal circumstances you will be running NAT on the Firewall (Network Address Translation) this is so you don’t deplete real internet addressing from your larger corporate addressing schema.

    IF this is the case then we want to make sure that traffic between 192.168.10.0/24 and 192.168.100.0 does not get translated, to accomplish this we will configure a NAT Exception

    same-security-traffic permit inter-interface
    same-security-traffic permit intra-interface
    no object-group-search access-control
    object network LAN
    subnet 192.168.10.0 255.255.255.0
    object network VPN_POOL
    subnet 192.168.100.0 255.255.255.0
    access-group allow-all global

    nat (inside,outside) source static LAN LAN destination static VPN_POOL VPN_POOL

    We created two network objects, one for our local network (LAN) and another one for the remote VPN users (VPN_POOL). The NAT rule tells the ASA not to translate traffic between the two networks.

    6. Split Tunnel: Not necessarily a good thing, when a user connects to the VPN, all traffic is forwarded through the VPN (ASA) to its destination, you can allow them access to the internet for browsing even when their VPN is connected or you can enable Split tunneling allowing the user to send only certain traffic through the VPN and the rest out their local internet connection. To do this we will create an access-list allowing the user to perform this action for traffic outside the know VPN range

    ASA0(config)# access-list SPLIT_TUNNEL standard permit 192.168.10.0 255.255.255.0

    7. Next lets create a group policy, this allows you to assign different remote users to different groups with different attributes. You might want to have a group policy for ‚Äúnetwork administrators‚Äù and another one for ‚Äúgeneral users‚Äù each with different DNS servers, timeout settings, etc

    group-policy VPN_POLICY internal
    group-policy VPN_POLICY attributes
    dns-server value 8.8.4.4
    vpn-idle-timeout 30
    split-tunnel-policy tunnelspecified
    split-tunnel-network-list value SPLIT_TUNNEL
    dynamic-access-policy-record DfltAccessPolicy

    8. Our group policy is called VPN_POLICY and it is an internal group policy which means it is created locally on the ASA. You can also specify an external group policy on a RADIUS server. We added attributes, for DNS and an idle timeout (30 minutes). Split tunneling refers to the access-list we created earlier.

    Next lets configure a user for remote access

    ASA0(config)# username VPN_USER1 password SECRET_PASS

    username VPN_USER1 password * pbkdf2

    9. Now we move on to creating the IPsec Settings with Phase 1

    ASA0(config-ikev1-policy)# authentication pre-share
    ASA0(config-ikev1-policy)# group 2
    WARNING: DH group2 configuration under crypto ikev1 policy group is insecure. Converted to DH group14. Please check release notes for details.
    ASA0(config-ikev1-policy)# lifetime 86400

    crypto ikev1 policy 10
    authentication pre-share
    encryption aes
    hash sha
    group 14
    lifetime 86400
    telnet timeout 5
    ssh key-exchange group dh-group14-sha256

    We are using AES for encryption and SHA for integrity, a pre-shared key and DH group 14 for key exchange, the lifetime before we have to renegotiate is 86500 (notice the message we received)

    10. Next we will enable IKEv1 policy on the outside interface

    ASA0(config)# crypto ikev1 enable outside

    ASA0(config)# crypto isakmp identity address

    11. Lets get IPSec Phase 2 going here

    ASA0(config)# crypto ipsec ikev1 transform-set TRANSFORM_SET esp-aes esp-sha-hmac

    that should look familiar as we did this in Lab 1 on the routers if you recall.

    12. We just configured a transform set called “TRANSFORM_SET” and its using ESP with AES/SHA, Next we will configure a Dynamic Crypto Map since its highly likely the connecting VPN user is behind dynamic addressing and we ‘wont’ be aware of which ones.

    ASA0(config)# crypto dynamic-map DYNAMIC_MAP 10 set ikev1 transform-set TRANSFORM_SET

    As you can see I named my dynamic map “DYNAMIC_MAP and it referes to my TRANSFORM_SET so even though we refer to it, we still need to attach the Static Crypto Map

    ASA0(config)# crypto map CRYPTO_MAP 10 ipsec-isakmp dynamic DYNAMIC_MAP

    13. next we will attach it to the outside interface

    ASA0(config)# crypto map CRYPTO_MAP interface outside

    Our final step is to create a tunnel group that will bind our group policy and pool together.

    ASA0(config)# tunnel-group TUNNEL_GROUP type remote-access

    ASA0(config)# tunnel-group TUNNEL_GROUP general-attributes

    ASA0(config-tunnel-general)# address-pool VPN_POOL

    ASA0(config-tunnel-general)# default-group-policy VPN_POLICY

    14. As you can see we named our tunnel group -TUNNEL_GROUP and we added the pool and group policy. Next we will configure the rest of its attributes, including the pre-shared key we will call SHARED_KEY

    ASA0(config-tunnel-general)# tunnel-group TUNNEL_GROUP ipsec-attributes
    ASA0(config-tunnel-ipsec)# ikev1 pre-shared-key SHARED_KEY

    This concludes the configurations on the ASA – next we move to the client.

    NOTE: You will need to place the underlying VMWARE NIC Card into promiscuous mode to be able to use an external “bridge” connector in order to reach the outside world from CML2.
